WKX Platform · Milestone 4 · For review

Observability

Every log a Service or the Host produces lands in CloudWatch under /wkx/, host metrics flow to the CWAgent namespace, one dashboard shows the box's vital signs and per-service request rate, and four host alarm types (disk, memory, CPU usage, CPU credit) email you via SNS. Billing is guarded by the wkx-org-monthly budget in infra/mgmt/. All of M4 is Terraform-managed in infra/aws/, platform account only.

Section 1 / 4

Scope and shape

M4 gives the platform eyes. Responsibilities split cleanly by layer, and each piece does only what it is naturally good at:

  • Docker daemon (Layer 2) ships container logs straight to per-service log groups via the awslogs log driver, authenticated by the instance role. Dual logging (Docker's default since 20.10) keeps docker logs working on the box.
  • CloudWatch agent (Layer 2, host deb, GPG-verified) handles host-level concerns only: syslog shipping and CPU, memory, disk, and network metrics.
  • CloudWatch metric filters derive request-rate metrics from Caddy's access logs server-side. Nothing extra runs on the box.

Spec correction

Design spec §4 lists the CloudWatch agent as a Layer 3 platform service in platform/compose.yml. Reality since M2 is a host deb installed by cloud-init, and the M4 GPG deliverable assumes it stays one. M4 corrects the spec: the agent is Layer 2 (host bootstrap). The correction ripples to the CONTEXT.md "Platform services" entry and is ADR material for the grill.

Roadmap amendment

Two lines change. "Docker container logs (JSON file driver)" becomes the awslogs driver: Docker's JSON log files live under content-addressed container paths, so the agent's path-to-log-group mapping cannot route them per service without a fragile deploy-time shim. And syslog's /wkx/system/<env> becomes /wkx/platform/<env>: "system" is a token that appears nowhere in the glossary, while "platform" already occupies the service slot without naming a Service (the platform-<env> Compose project), and the tagging strategy already treats "platform" as the shared category.

Out of scope: the home server (M9 decides what observability at home means; cloud-only config stays out of the base compose files so home stays clean), log-based alerting on app errors, tracing, and any third-party observability stack.

Section 2 / 4

Log pipeline

Log groups, created by Terraform

Log groupSourceRetentionTags
/wkx/hello/prod hello container via awslogs 7 days Env=prod, Service=hello
/wkx/caddy/prod Caddy runtime + JSON access logs via awslogs 30 days Env=prod, Service=caddy
/wkx/platform/prod Host syslog via CloudWatch agent 7 days Env=prod

Caddy's 30-day tier exists because access logs feed the request-rate metric and answer traffic questions after the fact. Streams are named by container via the driver's tag: "{{.Name}}" option.

Compose overlay, not base-file changes

The logging stanza is cloud-only, so it lives in a compose.cloud.yml overlay beside each compose.yml (platform/ and hello/). Cloud deploys run:

docker compose -f compose.yml -f compose.cloud.yml -p hello-prod up -d

The overlay adds, per Compose service:

logging:
  driver: awslogs
  options:
    awslogs-region: ap-southeast-2
    awslogs-group: /wkx/hello/prod
    awslogs-create-group: "false"
    tag: "{{.Name}}"
    mode: non-blocking
    max-buffer-size: "4m"

Base compose files stay driver-free, so the home server (M9) simply never applies the overlay. The reference project (M8) inherits the same pair, making the overlay part of the platform contract.

Caddy access logs and the request-rate metric

  • The wildcard site block in the platform Caddyfile gains a named access logger (log wkx), JSON format, written to stdout, so it rides the same awslogs pipeline into /wkx/caddy/prod. Verified on the deployed v2.11.4: one wildcard-block logger covers the imported snippet hosts and the 404 fallthrough.
  • A metric filter on /wkx/caddy/prod exact-matches $.logger = "http.log.access.wkx" (unnamed Caddyfile loggers get shifting auto-generated names) and extracts $.request.host as the Host dimension, publishing RequestCount to the WKX/Edge namespace.
  • Per-service series come straight off the host dimension; the platform total is metric math (SUM) in the dashboard widget. Requests that fall through to the wildcard 404 appear under their own host value, which makes probe traffic visible rather than hidden.

Section 3 / 4

Host changes

GPG-verified agent install

cloud-init currently curls the agent deb over HTTPS and installs it unverified. M4 replaces that with: download the deb and its .sig, import Amazon's published signing key, check the fingerprint is exactly 9376 16F3 450B 7D80 6CBD 9725 D581 6730 3B78 9C72, gpg --verify, and only then dpkg -i. A verification failure aborts bootstrap loudly rather than installing anyway.

Agent configuration via SSM Parameter Store

  • The agent config JSON lives in the repo at host/cloudwatch-agent.json: syslog collection into /wkx/platform/prod, plus aggregate CPU (usage_active), memory (used_percent), disk (used_percent on / and /srv/data, drop_device on), and network bytes on the primary interface only. Six custom metrics, dimensioned by InstanceId; Terraform re-points the alarms when the Host is replaced. The namespace stays CWAgent, matching the IAM condition pinned in M2.
  • Terraform publishes the file to an SSM parameter (proposed: /wkx/platform/prod/CLOUDWATCH_AGENT_CONFIG, inside the instance role's existing /wkx/* read scope).
  • At boot, cloud-init runs amazon-cloudwatch-agent-ctl -a fetch-config -c ssm:<param> -s. Config changes later are a Terraform apply plus an SSM RunCommand re-fetch, with no host replacement.
Planned host replacement

Changing cloud-init replaces the Host (ADR 0017). M4 therefore includes one planned instance replacement, done first so everything else lands on the new box. The Data volume and Caddy's certificates survive; the platform stack and hello are redeployed after, per the M3 state doc. The replacement drill was exercised in M2.

Section 4 / 4

Alarms, dashboard, and repo changes

SNS and email

One wkx-alerts topic in ap-southeast-2 with an email subscription. The address is the alert_email Terraform variable with no default; the real value lives in gitignored terraform.local.tfvars (invariant 7), the same pattern infra/mgmt/ already uses. No CloudWatch billing alarm: the wallet guard is the wkx-org-monthly budget (80% actual / 100% forecasted of USD 45), which avoids the irreversible payer-account billing-alerts preference and cross-region SNS plumbing.

The alarms

AlarmMetricThreshold
Disk ×2 disk_used_percent on / and /srv/data > 80% for 15 min
Memory mem_used_percent > 90% for 15 min
CPU usage cpu_usage_active > 80% for 15 min
CPU credit CPUCreditBalance (AWS/EC2, free) < 144 for 15 min (25% of the 576-credit bank)

Thresholds are deliberately conservative to avoid noise; they are one-line Terraform changes to tune. The Host runs standard CPU credits (ec2.tf), so an empty bank throttles to the 20%-per-vCPU baseline rather than billing: the credit alarm predicts that cliff, the usage alarm warns the bank is draining. Five alarms, within the ten free.

Dashboard

One Terraform-managed CloudWatch dashboard: CPU, memory, disk, network, and request rate (per-service series stacked, with the total). Widgets read from CWAgent and WKX/Edge.

Terraform and repo layout

  • infra/aws/ gains logs.tf, cloudwatch_agent.tf, sns.tf, alarms.tf, dashboard.tf (file-per-concern, matching the root's existing style); variables.tf gains alert_email with no default.
  • terraform test invariants extend to: every log group has a retention policy and the right tags, every alarm has an action wired, the dashboard exists.
  • Compose overlays: platform/compose.cloud.yml, hello/compose.cloud.yml; the Caddyfile gains the access-log directive.
  • cloud-init: GPG verification plus agent bootstrap from SSM.
  • Docs: docs/setup/m4-infra-state.md (+ gitignored .local.md), the roadmap amendment, spec §4 correction, CONTEXT.md glossary update, and ADR candidates for the awslogs decision and the agent's layer. The grill pass owns the last two.

Verification (hands-on artefacts)

  1. Tail Caddy and hello logs live in the CloudWatch console (roadmap artefact).
  2. aws cloudwatch set-alarm-state forces a host alarm; the email arrives (roadmap artefact, amended).
  3. A short curl loop against hello.wingkongexchange.dev moves the request-rate widget.
  4. Agent status is healthy via SSM; the GPG verification shows in the cloud-init log.

Cost

Effectively zero at current scale: the always-free tier covers the ~1 GB log ingest (5 GB free), the custom metrics (10 free), the five alarms (10 free), and the one dashboard (3 free). Costs appear as the fleet grows, at USD 0.30 per metric and USD 0.50 per GB in the first paid tiers.
